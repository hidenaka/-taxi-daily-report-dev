# 計画書: 現在地ベース 推奨検索（A案）

> 2026-05-20 起票。本人指摘「`support.html` の推奨検索の候補が乗車場所しか出ない、現在地ベース or 過去降車パターンのどちらかに寄せたい」→ **A案（現在地ベース）採用**で確定。
> 既存の「降ろした場所 × 時刻 → 次に行くべきエリア」モデル（B案）は廃止しない。**プルダウンの候補ソース** だけ「現在地周辺」に拡張する。

## ゴール

**「今いる場所」から推奨検索ができるようにする。**
- GPS で取得した現在地のエリアを **第一候補** にする（既存）
- それを起点に、**距離が近いエリアを候補として並べる**（新規）
- 過去に降車実績のないエリアでも候補に出る（新規）

## 現状（変更前）

`support.html` の「次の営業先 推奨検索」プルダウン `#recArea` の候補:
- `filterRecAreaCandidates(dropoffAreaAnalysis(allDrives), OFFICE_AREA)` で生成
- ＝ **「過去に自分または全員が降車したエリア」のみ**（件数降順）
- 新規エリアは選択肢にない → 引越し直後/出張先で使えない

選択時の初期値:
- `chooseInitialRecArea` で「GPS → localStorage → 会社設定 → 丸の内」の優先順位
- ただし候補リストに無いエリアは選択不可

## 目標設計（変更後）

### プルダウン候補ソース
1. **現在地が取れている場合**: 現在地エリアを最上位、距離順に近いエリアを上位に
2. **取れていない場合**: 過去降車実績エリア（既存挙動を維持）
3. どちらにも該当しない極端なケース: 都内主要エリア（千代田区丸の内、新宿区西新宿、渋谷区桜丘町 等）の固定リスト

### 距離計算
- エリア = 「区+町名」（既存粒度）
- 各エリアに代表座標を持たせる（**新規実装が必要**）
- ハバシン公式で「現在地 → エリア代表座標」を計算 → 近い順にソート

### UI 変更
- プルダウン内に「📍 現在地周辺」「📊 過去降車実績」のグループラベル
- GPS取得済時:
  - 上位5件 = 現在地周辺（距離順）
  - その下 = 過去降車実績（件数順）
- GPS未取得時:
  - 既存挙動と同じ（過去降車実績のみ）

## 必要な実装

- スキーマ: `{ "千代田区丸の内": { "lat": 35.681, "lng": 139.767 }, ... }`
- データソース: 既存 GeoJSON + 国土地理院 API or 手動キュレーション

- `pickNearbyAreas(centroids, fromLat, fromLng, limit=10): Area[]` — 上位N件

## リスク

| リスク | 影響 | 対策 |
|---|---|---|
| エリア代表座標が不正確 | 距離計算がズレ、推奨順が悪い | 主要エリアから手動キュレーション、低頻度エリアはfallback (区中心座標) |
| GPS精度（屋内/トンネル） | 異常な近傍エリアが出る | 既存 `recGps` の精度バリデーション流用 |
| 過去降車実績との重複 | 同じエリアが2回出る | de-duplication で1回に集約 |
| エリア未登録の現在地 | 「近隣推奨」が空 | fallback で過去実績モードに自動切替 |

## 既存 B案（降車パターン→次乗車）との関係

- **B案の集計ロジック自体は維持**（既存検索結果の品質を落とさない）
- **入力候補の幅を広げる** = ユーザーが現在地起点で「ここから次へ」と入力できるようになるだけ
- 過去実績がないエリアを選んだ場合は、既存の「近隣エリアも含める」フォールバックを活用
