# 会社別プロファイル（マルチカンパニー）設計

作成: 2026-05-18 ／ 状態: ドラフト（ユーザーレビュー待ち）

## 概要

タクシー会社ごとに歩率・給与・各種条件が異なる。現在アプリの計算は
`js/default-config.js` の `DEFAULT_CONFIG`＝中野氏の会社（恵豊）の設定に固定されて
おり、他社のドライバーが登録すると誤った歩率表で計算され数値が全てずれる。

「会社」を独立した**会社プロファイル**として持ち、会社ごとの申込リンクで
正しいプロファイルを適用する仕組みを導入する。

## 背景・経緯

- 課金システム（C）・アカウント（B）・動線最適化まで dev 実装済み。
- registration を恵豊以外に開ける前に本機能が必要（2026-05-18 ユーザー判断）。
- 提携（組合）価格が安い根拠＝**組合／会社が自社の正確な条件（部立・休憩・
  高速代・車両種類等）を取りまとめて提供する作業を担う**こと。通常プランは
  その作業を中野氏側が負う（または本人が自己設定する）ため正価。

## 設計・方針

### 1. 会社プロファイル = 独立エンティティ

Firestore に `companies/{companyId}` コレクションを新設。各社1ドキュメント。

- メタ: `name`, `slug`（申込リンク用）, `active`, `plan`（partner / normal）
- **会社レベル設定**（その会社のルール）:
  - 部立／歩率テーブル（乗務数別の売上ティアと歩率）
  - 休憩時間ルール
  - 会社負担の高速代
  - 車両種類（その会社で選べる乗務種別）
  - 給与ルール（手取り率・有給手当・12〜13出番の固定率特例 等）
- 価格: 適用する Stripe の価格／プロモーションコード

### 2. 参照モデル（コピーしない）

会社プロファイルを**唯一の正**とする。各ユーザーの計算は実行時に
「会社プロファイル ＋ 本人の個人設定」をマージして行う。中野氏が会社
プロファイルを更新すれば、その会社の全ドライバーに即反映される。

| レベル | 保存先 | 内容 |
|---|---|---|
| 会社レベル | `companies/{companyId}` | 部立・休憩・高速代・車両種類・給与ルール・価格 |
| 個人レベル | `userConfigs/{userId}` | 手取り目標・総支給目標・表示名・天候地点 等 |

### 3. 会社ごとの申込リンク

`?company=<slug>` を任意のページ読込で捕捉し localStorage に保持、登録時に
`users/{uid}.companyId` へ確定する。

- **提携会社のドライバー**: 会社が配布する固有リンク（例
  `app.taxicabis.com/?company=keiho`）から登録 → 会社プロファイル適用 →
  申込時にその会社の提携価格を自動適用（手動の割引コード入力は不要）。
- **通常プラン（非提携）**: 会社リンク無しで登録 → 通常 1,000円。歩率等は
  本人が自己設定、または中野氏の構築代行を相談（運用判断）。

### 4. 恵豊のパッケージ化

現 `DEFAULT_CONFIG` の会社レベル項目を抽出して `companies/keiho` プロファイル
を作成（現挙動と等価）。既存2ユーザー（`user_self`, `mm`）に
`companyId = keiho` を付与する。

### 5. 管理者

`admin.html` に「会社管理」を追加。中野氏が各社のプロファイル（部立・休憩・
高速代・車両種類・給与ルール・価格）を作成・編集し、申込リンクを発行する。
会社を増やすたびにこの操作を行う。

## 詳細

### データ
- `companies/{companyId}`: 上記フィールド。
- `users/{uid}` に `companyId` を追加。
- `firestore.rules`: `companies/{companyId}` は read = 該当会社所属ユーザー
  または admin、write = admin のみ。
- `userConfigs/{userId}` は個人レベル項目のみ保持に縮小（会社レベル項目は
  会社プロファイル参照に移行）。

### 設定マージ
- `js/storage.js` の `getConfig()` を「会社プロファイル＋userConfig のマージ」に
  改修。会社レベル項目は会社プロファイルを優先。
- `DEFAULT_CONFIG` は「会社未設定時のフォールバック（通常プランの初期値）」
  として残す。

### 申込リンク捕捉
- index / login / subscribe の各ページで `?company=` を読み、妥当なら
  localStorage `taxi_pending_company` に保存。登録（signUp）時に consume し
  `users/{uid}.companyId` に書き込む。
- 既存の `?user=` パラメータ処理と同じ場所に追加。

### 価格
- subscribe.html は `companyId` から価格を決定（手動クーポン欄は廃止、または
  会社リンク無しユーザー向けの任意入力として残す）。
- Worker `/create-checkout-session` は `companyId` を受け取り、該当会社の
  プロモーションコードを適用する。

### 通常プラン（自己設定）
- `companyId` 無しのユーザーは「自分専用の編集可能なプロファイル」を持ち、
  歩率テーブル等を本人が入力できる（現在 admin 限定の rateTable エディタを
  当該ユーザーに開放）。構築代行＝中野氏がこの入力を代行するだけで、
  別機能ではない。

## 段階（実装順）

1. データモデル＋恵豊プロファイル化（既存挙動を壊さず恵豊＝現 DEFAULT_CONFIG と等価に）
2. 会社別申込リンク＋登録時 `companyId` 付与
3. admin 会社管理UI
4. subscribe の会社別価格の自動適用
5. 通常プランの自己設定UI

各段階を dev で検証してから次へ進む。

## 残論点（実装中／レビューで詰める）

- 通常プラン自己設定で、一般ユーザーに歩率テーブルエディタをどこまで開くか。
- 会社プロファイル更新時、ユーザーが個別調整した値との衝突ポリシー
  （会社レベル項目は個人で上書き不可とする想定）。
- 会社ごとに価格が異なる場合の特商法表示（専門家確認論点）。
- 既存ユーザーが登録済みで `companyId` 未設定のケースの移行手順。
